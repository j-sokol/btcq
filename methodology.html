BTCQ / Methodology

Scoring rules and privacy boundary

BTCQ is a heuristic classifier built on top of public Esplora API data. It reports practical exposure signals, not a formal proof of cryptographic safety.

Core inputs

  1. Address summary stats from `/address/:address`.
  2. Recent and paged confirmed history from address transaction endpoints.
  3. Output script type plus spend-path hints from `vin` and `vout` data.

Back to checker

Risk tiers

  • Tier 1: `P2PK` outputs publish a public key directly in the locking script.
  • Tier 2: Spent `P2PKH`, `P2WPKH`, `P2WSH`, or spent `P2SH` with revealed redeem or witness paths.
  • Tier 3: `P2TR` output keys are committed on-chain even before spend-path details are known.
  • Tier 4: Common `P2PKH` or `P2WPKH` addresses with no detected spend history.
  • Tier 5: Unknown or currently under-specified cases, including unspent `P2SH` and no-history addresses.

Complex script handling

  • Spent `P2SH` is no longer treated as generically unknown if the API reveals a redeem or witness script path.
  • Wrapped SegWit spends are flagged when the redeem script begins with `OP_0 ...`.
  • Multisig is flagged when revealed script assembly includes `CHECKMULTISIG`.
  • `P2WSH` spends are treated as script-path exposure because witness-script details are typically revealed at spend time.

Reuse signal

BTCQ flags likely address reuse when spent output count exceeds zero. An address that has only received — but never sent — is not flagged, because key material is only revealed on the spend path. This is a practical signal, not a wallet-descriptor analysis.

Privacy boundary

BTCQ has no backend and does not proxy requests, but the selected public API provider still sees the lookup. Stronger privacy requires a self-hosted endpoint or a separate network-privacy layer.

Known limits

BTCQ does not parse every possible custom script or establish cryptographic certainty. It is an educational triage tool built from explorer-visible data.
